# 店策 Agent：抖店 × 巨量千川经营副驾

[![Version](https://img.shields.io/badge/version-2.5.3-2563eb)](extension/manifest.json)
[![Chrome](https://img.shields.io/badge/Chrome-Manifest_V3-16a34a)](extension/manifest.json)
[![License](https://img.shields.io/badge/license-MIT-f59e0b)](LICENSE)
[![Mode](https://img.shields.io/badge/data-local--first-7c3aed)](SECURITY.md)

**不申请开发者 API，也能把抖店和巨量千川网页数据变成运营每天真正能执行的任务。**

店策 Agent 是一个本地优先、只读的 Chrome 扩展与 AI Agent 数据桥。它复用运营人员已经登录的抖店、巨量千川网页，在用户主动点击后采集可见经营数据，完成脱敏、本地保存、异常诊断，并生成千川调整建议、库存预警、直播/货架分析和每日经营报告。

> 当前版本：`v2.5.3 Beta`。本地 Agent 可随 Windows 登录自动启动；数据巡查仍仅由用户点击触发，不会每 5 分钟后台自动运行，也不会自动修改预算、库存或计划状态。

## 为什么做这个项目

很多店铺没有开放平台 API 权限，但运营仍要每天在多个后台页面之间反复切换、复制数据、判断风险和写报告。常见问题包括：

- 数据散落在抖店、千川、精选联盟等不同页面，查看成本高。
- 运营看到指标，却不知道今天最该先处理哪三件事。
- 千川计划调整依赖经验，缺少统一的止损、观察和放量口径。
- 库存、投放、直播和货架各自优化，容易出现“流量放大但库存断货”。
- 日报花时间整理数字，却很难形成负责人、动作和验收标准。

店策 Agent 的目标不是再做一块复杂看板，而是把数据收拢成一个清晰闭环：

```text
用户点击巡查 → 网页采集与脱敏 → 本地快照与趋势 → 风险诊断
          → 今日三件事 → 分配负责人 → 执行 → 复盘验收
```

## 谁适合使用

| 角色 | 最关心的问题 | 店策 Agent 提供什么 |
| --- | --- | --- |
| 运营总管 | 今天先处理什么、谁负责、何时验收 | 跨岗位任务排序、负责人、证据和验收标准 |
| 千川投手 | 哪些计划止损、观察、换素材或放量 | 计划级建议、判断依据和安全边界 |
| 直播运营 | 直播间流量为什么没有转化 | 观看→商品点击→成交漏斗诊断 |
| 货架运营 | 曝光、点击、成交卡在哪一层 | 货架漏斗、搜索/推荐卡诊断和优化动作 |
| 商品运营 | 哪些商品缺货、库存还能卖几天 | 缺货、极低库存、低库存和可售天数预警 |
| 店铺负责人 | 每日经营结果和异常是否闭环 | 自动生成本地日报、趋势对比和任务状态 |

## 核心能力

### 1. 无 API 的手动全店巡查

- 复用用户已登录的抖店、千川和精选联盟网页，不读取 Cookie。
- 仅在点击“全店巡查”后运行，依次打开最多 16 类核心页面。
- 巡查使用独立后台标签页，不操作用户原有页面；结束后自动关闭。
- 对列表页尝试虚拟滚动并最多采集 5 页，同时记录行数、滚动次数和截断提示。
- 逐页记录成功、失败、数据质量和页面类型；失败页面自动重试一次。

### 2. 经营数据自动填写与本地历史

- 按 `数据源 + 页面类型` 保存快照，订单、商品、库存和计划数据不会互相覆盖。
- 默认保留 30 天脱敏历史，展示近 7 天核心指标变化。
- URL 自动移除查询参数和 hash，页面原始 DOM 不会落盘。
- 数据只发送到本机 `127.0.0.1:8765`，不依赖云端数据库。

### 3. 千川计划调整方案

- 按计划明细给出止损、保持观察、素材优化和谨慎放量建议。
- 结合 ROI 目标、消耗门槛和转化表现展示判断依据。
- 建议带安全边界，所有预算和启停操作仍由投手人工确认。

### 4. 库存预警

- 识别缺货、极低库存和低库存商品。
- 在数据允许时估算预计可售天数。
- 把库存风险与直播、货架和投放任务放在同一优先级队列中。

### 5. 直播与货架数据分析

- 分析店铺直播和千川直播大屏的观看→商品点击→成交漏斗。
- 分析货架曝光→点击→成交漏斗。
- 输出风险解释、建议动作、负责人和复查指标，而不只展示数字。

### 6. 运营总管与每日经营报告

- 总管、货架、直播、投放、商品五种岗位视图。
- 首页最多展示 3 个当日必做事项，降低信息负担。
- 任务支持待处理、进行中、待观察和已完成状态。
- 可立即生成 Markdown 日报，也可按设定时间使用最近一次缓存数据生成日报。
- **定时日报不会触发网页巡查**；需要最新数据时，运营先手动点击巡查。

## 产品原则与安全边界

| 原则 | 当前实现 |
| --- | --- |
| 用户掌控 | 所有网页巡查均由用户点击触发 |
| 默认只读 | 不修改库存、不发货、不回复客户、不调整预算、不启停计划 |
| 最少采集 | 只读取经营指标和允许的表格字段，不保存整页 DOM |
| 本地优先 | 快照、历史、任务和报告保存在用户电脑 |
| 建议可解释 | 每个建议尽量包含证据、动作、安全边界和验收指标 |
| 人工确认 | 涉及资金、库存和账号状态的操作必须由运营在官方后台完成 |

详细安全说明见 [SECURITY.md](SECURITY.md)。

## 快速开始

### 1. 下载项目

```bash
git clone https://github.com/3132882323-cyber/douyin-agent.git
cd douyin-agent
```

也可以在 GitHub 页面点击 `Code → Download ZIP` 后解压。

### 2. 安装 Chrome 扩展

Chrome 不允许普通网页直接安装未上架扩展，开发版需要“加载已解压的扩展程序”：

1. 打开 `chrome://extensions/`。
2. 开启右上角“开发者模式”。
3. 点击“加载已解压的扩展程序”。
4. 选择项目中的 `extension` 文件夹，不要选择仓库根目录或 ZIP 文件。
5. 将“店策 Agent”固定到工具栏。

安装成功后，扩展详情应显示版本 `2.5.3`。

### 3. 启用本地 Agent 自动启动

Windows 用户首次使用时双击：

```text
bridge/enable_autostart.bat
```

脚本会创建独立的 Python 运行环境、安装依赖、立即启动本地 Agent，并将其加入当前用户的 Windows 登录启动项。以后登录 Windows 后无需再次手动启动。

随后访问 <http://127.0.0.1:8765/health>。看到 `"status": "ok"` 即表示本地服务正常。需要临时手动启动时仍可运行 `bridge/start_bridge.bat`；需要关闭自动启动时运行 `bridge/disable_autostart.bat`。

> 抖店和千川账号登录由 Chrome 官方登录页面及浏览器登录状态管理。本项目不会保存账号密码，也不会绕过验证码或平台风控。

### 4. 开始第一次巡查

1. 在 Chrome 中登录抖店和巨量千川。
2. 点击扩展图标，确认网页数据源与本地 Agent 状态正常。
3. 点击“全店巡查”获取多页面数据，或在当前页面点击“立即同步”。
4. 打开“经营副驾”，查看今日三件事、千川建议、库存风险和直播/货架分析。
5. 在策略设置中填写 ROI 目标、消耗判断门槛和低库存阈值。

## MCP 接入

本地 Agent 可以把已脱敏数据和诊断结果提供给支持 MCP 的 AI 客户端。

```json
{
  "mcpServers": {
    "dian-agent": {
      "command": "python",
      "args": [
        "C:\\path\\to\\douyin-agent\\bridge\\server.py"
      ]
    }
  }
}
```

## 已知限制

- 网页字段没有官方稳定契约，平台改版后可能需要更新选择器。
- 虚拟滚动表格只能读取浏览器实际渲染过的行，分页采集也设有上限。
- 登录过期、验证码和平台风控必须由用户手动处理。
- 网页经营数据不会定时抓取；所有巡查都需要用户点击触发。
- 定时日报需要本地 Agent 保持运行，并使用最近一次缓存数据。
- 当前诊断以确定性规则为主，建议执行前应回到官方后台核对口径。
